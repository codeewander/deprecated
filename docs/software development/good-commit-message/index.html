如何寫好的 Commit Message

什麼是 Commit Message ?#

  • Commit message 是程式碼變更及其變更原因的公開紀錄，該紀錄是永久的。
  • Commit message 最好兼具 Why 和 What，方便日後維護人員更快進入狀況

Commit Message 規範#

Commit message 包含三個部分：

1. Header: <type>(<scope>): <subject>#

  • type: 代表 commit 的類別：feat、 fix、 docs、 style、 refactor、test、 chore 等，由團隊自行定義
  • scope: 代表 commit 影響的範圍，例如哪個 Components 等，視專案不同而異
  • subject: 代表此 commit 的簡短描述，需要清晰、直指重點，使用命令句(祈使句)，不要超過 50 個字元，使結尾不要加句號

Header 範例：不好的寫法 => 較好的寫法

  • 沒有說明錯誤和如何修復

    fix: Modify typo => fix: Modify typo(displya => display)

  • 沒有使用命令句

    chore: Updating languageBar package version from 1.0.1 to 1.0.2 => chore: Update languageBar package version from 1.0.1 to 1.0.2

常見 Type 類別說明

  • feat: 新增/修改功能 (feature)
  • fix: 修補 bug (bug fix)
  • docs: 文件 (documentation)
  • style: 格式 (不影響程式碼運行的變動 white-space、formatting 等)
  • refactor: 重構
  • perf: 改善效能
  • test: 測試
  • chore: 建構程序或輔助工具的變動 (如：更新版本、CICD 變動等)

Type 可以協助 Reviewer 使用不同角度來檢視 Commit 內容，可以提升 Code Review 的速度，但前提是開發團隊對於 Type 有一定的共識。 例如：

  • 看到 Type 為 fix，Reviewer 可以用「如何修復錯誤」的角度來閱讀程式碼。

2. 身體 Body#

  • 描述更詳細的資訊，包括欲解決的問題、為何使用此解法、此解法的優缺點等
  • 倘若相關，也可附上相關設計文件、任務編號、效能測試結果等

3. 結尾 Footer#

  • 若有的話可放上任務編號

好用工具#

  • Gitmoji: github commit 加上表情符號
  • Conventional Commits: commit 提交工具
  • commitlint：commit 提交檢查工具
  • commitlint-jira: 結合 Jira 的 commit 規範

Commit 觀念#

  1. 一個 commit 基本上處理一個改動，最好將任務拆分成多個小的 commit
  2. 推上遠端分支前，可以透過 git rebase <hash> -i 整理(編輯/合併/拆分) commit
  3. Merge 到主分支前若想讓主分支的 Commit 更乾淨，可以選擇 Merge Squash
