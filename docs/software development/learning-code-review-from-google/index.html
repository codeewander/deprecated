Code Review 心法

前陣子剛好整理跟 Google 的 Code Review Guide 並跟同事分享（簡報連結-跟 Google 學 Code Review)，趁這個機會也把一些重點整理在這裡。

為什麼要有 Code Review?#

Code Review 主要目的是改善程式碼品質，程式碼須符合良好的功能性以及品質原則才能通過(有安全性或法律問題等緊急情況之下以快速達到功能修復為主，程式碼品質為其次)。透過 Code Review，開發者間可以進行技術交流和知識轉移。然而，沒有「完美」的程式碼，倘若標準過高，則可能會導致開發進度延遲，且開發者可能降低意願去改善程式碼

Code Review 準則#

  • 事實數據 > 個人偏好

    按照既有的 style guide 或專案程式碼風格，而非個人偏好

  • 程式碼維持風格一致性

  • 當發生意見衝突時，建議面對面討論或視訊會議，若還是無法找到共識，則可邀請團隊共同參加討論，會議中應有技術主管等決策者，討論結果也可納入 style guide

Code Review 審查要項#

Design 設計#

  • 程式碼運行方式合理
  • 與既有專案的整合是否適當
  • 是否為正確的時間點加入此變動

Functionality 功能#

  • 此變動是否達到預期目標
  • 對使用者(含 users 和其他開發者)帶來益處
  • 考量極端案例，尋找併發性問題
  • 驗證修改，檢查會對使用者造成影響的行為
  • 確保沒有因平行運算而造成 deadlock 或 race-condition

Complexity 複雜度#

  • 閱讀程式碼者可以很快理解：考量未來維護，開發者不應向審核者說明程式碼，若不容易理解，則開發者應回去調整或加註釋，再請審核者進行審核。
  • 沒有過度設計

Test 測試#

  • 依據改動適當要求加入單元測試、整合測試或 end-to-end 測試
  • 相同改動必須加入測試(緊急狀況除外)，並視為產品的一部分
  • 確保測試是正確、合理且有用的

Naming 命名#

  • 良好的命名：明確表達且容易閱讀

Comments 註釋#

  • 解釋 why 而非 what
  • 註釋清楚好懂且有用處

Style 樣式#

  • 遵守 style guide，保持一致性
  • 若有不在 style guide 中的樣式建議，則加上「非必要」之類的文字

善用 linter 工具 ex: eslint 來解決樣式一致性的問題

Documentation 文件#

  • 撰寫清楚的文件

Every Line 審視每一行程式碼#

  • 可用不同審視角度減少盲點
  • 倘若審查者沒有查看每一行，記得標註已審視的部分

審查速度和流程#

  • 流程：架構 => 功能 => 程式碼細節
  • 回覆審查的速度應為一個工作天內(並非指整個審查時間)，倘若審查者正進行開發任務，不應被打斷，可建議其他審查者、給定明確審查時間或提供大方向的改動建議

如何撰寫審查建議#

  • 保持尊重和友善
  • 提供具體回饋及建議原因
  • 考量時間限制：若審查建議的執行時間需要很長，則考慮另開 ticket 處理
  • 區分必要和非必要改動
  • 看到好的程式碼，不吝於讚美和鼓勵
